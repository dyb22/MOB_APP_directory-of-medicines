# Полный цикл DevOps: справочник лекарств (стенд команды)

Документ описывает **инфраструктуру разработки и сопровождения**, развёрнутую в рамках учебного проекта: **Wiki**, **Gitea**, **Drone CI**, **Grafana** и **Prometheus**, а также связь с **Android-приложением** и **Git**.

---

## 1. Роли инструментов

| Компонент | Назначение |
|-----------|------------|
| **Gitea** | Собственный Git-сервер: репозитории, веб-интерфейс, интеграция с CI. |
| **Wiki.js** | Отдельная wiki для документации проекта, требований и отчётных материалов. |
| **Drone** | CI/CD: автоматический запуск пайплайна при push в репозиторий на Gitea. |
| **Drone Runner** | Исполнитель шагов пайплайна (Docker runner на той же ВМ). |
| **Prometheus** | Сбор и хранение метрик (мониторинг). |
| **Grafana** | Визуализация метрик, дашборды, Explore. |
| **Docker Compose** | Описание и запуск всех перечисленных сервисов из каталога `infra/`. |

**Разработка Android-приложения** ведётся **локально в Android Studio**; на ВМ развёрнут серверный DevOps-стек и при необходимости клон/рабочая копия репозитория.

---

## 2. Сетевая схема и доступ

- Доступ к учебной сети и ВМ обеспечивается согласно инструкции кафедры (VPN / WireGuard, Proxmox при необходимости).
- Пример адреса ВМ в локальной сети: **192.168.91.222** (фактический IP выдаётся для каждой группы).
- Сервисы доступны по HTTP с рабочей станции при нахождении в той же сети (или через VPN).

### Порты по умолчанию

| Сервис | Порт | URL (пример) |
|--------|------|----------------|
| Gitea | 3000 | `http://<IP_ВМ>:3000` |
| Gitea (Git по SSH) | 2222 | `ssh://git@<IP_ВМ>:2222/...` |
| Wiki.js | 3080 | `http://<IP_ВМ>:3080` |
| Drone | 9000 | `http://<IP_ВМ>:9000` |
| Grafana | 3001 | `http://<IP_ВМ>:3001` |
| Prometheus | 9090 | `http://<IP_ВМ>:9090` |

## 3. Подготовка ВМ

1. Установлена **Ubuntu Server** (в инструкции курса — 24.04 LTS).
2. Созданы пользователи группы, настроен **OpenSSH** для удалённой работы (VS Code Remote SSH и т.п.).
3. Установлен **Docker** и плагин **Docker Compose** из репозиториев Ubuntu.
4. Пользователь добавлен в группу `docker`; для применения группы в текущей сессии использовались `newgrp docker` или повторный вход по SSH.
5. Проверка: `docker run --rm hello-world`.

---

## 4. Репозиторий проекта и каталог `infra`

- Исходный код приложения и инфраструктура хранятся в одном Git-репозитории (модули `app`, `domain`, `data`, каталог **`infra/`**).
- В **`infra/`** находятся:
  - **`docker-compose.yml`** — описание сервисов Gitea, Wiki.js, Postgres для wiki, Drone, Drone Runner, Prometheus, Grafana;
  - **`prometheus/prometheus.yml`** — конфигурация сбора метрик Prometheus;
  - **`.env.example`** — шаблон переменных окружения;
  - **`.env`** — локальный файл на сервере (не коммитится): пароли, секреты Drone, `HOST_IP`, `GITEA_ROOT_URL` и т.д.

На ВМ репозиторий размещается, например, в `~/projects/MOB_APP_directory-of-medicines`.

---

## 5. Конфигурация окружения (`.env`)

В файле **`infra/.env`** задаются минимум:

- **`HOST_IP`** — IP ВМ, с которого открываются сервисы в браузере.
- **`GITEA_ROOT_URL`** — базовый URL Gitea, например `http://<IP>:3000/`.
- **`WIKIJS_DB_PASSWORD`** — пароль БД для Wiki.js (PostgreSQL в compose).
- **`GRAFANA_ADMIN_PASSWORD`** — пароль пользователя `admin` в Grafana.
- **`DRONE_RPC_SECRET`** — общий секрет между Drone Server и Runner (генерация: `openssl rand -hex 16`).
- **`DRONE_GITEA_CLIENT_ID`** и **`DRONE_GITEA_CLIENT_SECRET`** — данные OAuth2-приложения в Gitea для входа в Drone (Redirect URI: `http://<IP>:9000/login`).

Запуск стека:

```bash
cd ~/projects/MOB_APP_directory-of-medicines/infra
docker compose pull
docker compose up -d
docker compose ps
```

## 6. Gitea

1. Первичная настройка через веб-мастер по адресу Gitea.
2. В форме установки для корректных URL клонирования по SSH указан **внешний порт 2222** (проброс с контейнера на хост), а не 22.
3. Создана учётная запись пользователя; заведён репозиторий проекта (например **`st22_065/MOB_DIRECTORY`**).
4. С локальной машины добавлен второй remote **`gitea`** и выполнялся push в Gitea параллельно с GitHub (`origin`), чтобы код и `.drone.yml` были доступны CI.

## 7. Wiki.js

1. После запуска compose открыт адрес **`:3080`**, пройден мастер установки Wiki.js.
2. В wiki размещаются материалы проекта: описание приложения, требования, инфраструктура, ссылки на сервисы.
3. Текст настоящего документа может быть перенесён в Wiki.js как отдельная страница.

## 8. Drone CI

### 8.1. Файл пайплайна

В **корне** репозитория добавлен **`.drone.yml`** (имя с **точкой** в начале). В настройках репозитория в Drone поле **Configuration** указывает на `.drone.yml`.

Текущий пайплайн учебного уровня использует образ **Alpine** и проверяет работоспособность runner (команды `echo`, `uname`); при необходимости пайплайн можно расширить (например, сборкой Gradle при наличии образа с Android SDK).

### 8.2. Интеграция с Gitea

- В Gitea создано **OAuth2-приложение** для Drone с Redirect URI **`http://<IP_ВМ>:9000/login`**.
- Репозиторий **активирован** в Drone (отображается в списке, включены сборки).

### 8.3. Вебхук

- Первоначально вебхук на **`http://<IP_ВМ>:9000/hook`** с хоста Gitea мог быть недоступен из контейнера Gitea (политика SSRF / список разрешённых хостов).
- **Target URL** вебхука заменён на **`http://drone/hook?secret=...`** (обращение к сервису Drone по **внутреннему DNS Docker** на порт **80**).
- В **`data/gitea/gitea/conf/app.ini`** добавлена секция:

```ini
[webhook]
ALLOWED_HOST_LIST = *
```

(для учебного стенда; в продакшене список ограничивают точнее).

После этого доставки вебхуков стали **успешными**, при push запускаются сборки в Drone.

### 8.4. Результат

- В интерфейсе Drone отображаются **успешные сборки** (например, Build #1 по коммиту в `master`).
- Скриншоты успешного пайплайна используются в отчёте.

## 9. Prometheus и Grafana

### 9.1. Prometheus

- Контейнер **Prometheus** публикует метрики на порту **9090**.
- Для записи данных в примонтированный каталог на хосте в **`docker-compose.yml`** для сервиса Prometheus (и при необходимости Grafana) задано **`user: "0:0"`**, чтобы избежать ошибок прав доступа к каталогам `data/prometheus` и `data/grafana`.

### 9.2. Grafana

- Вход: пользователь **`admin`**, пароль из **`GRAFANA_ADMIN_PASSWORD`**.
- Добавлен **источник данных Prometheus** с URL **`http://<IP_ВМ>:9090`**; проверка **Save & test** завершилась успешно.
- В разделе **Explore** выполняются запросы PromQL, например:
  - **`up`** — доступность целей scrape;
  - **`process_resident_memory_bytes`**, **`rate(process_cpu_seconds_total[1m])`** — внутренние метрики процесса Prometheus без дополнительных экспортеров.

При расширении стенда в Prometheus добавляют scrape для **node_exporter** и других экспортеров для метрик ОС и приложений.

## 10. Работа с Git в «полном цикле»

1. Разработка и коммиты **локально** (Android Studio / терминал).
2. **Push** в **GitHub** (`origin`) — для общей работы команды и резервной копии.
3. **Push** в **Gitea** (`gitea`) — чтобы на ВМ был актуальный код и срабатывал **Drone**.
4. На ВМ выполняется **`git pull`** с нужного remote для обновления файлов и перезапуска compose при изменении `infra/`.

## 13. Замечания по безопасности

- Файл **`infra/.env`** содержит секреты и **не должен** попадать в публичный Git (учтено в `.gitignore`).
- Значение **`ALLOWED_HOST_LIST = *`** для вебхуков Gitea приемлемо для учебной изолированной сети; в реальных системах список хостов ограничивают.
